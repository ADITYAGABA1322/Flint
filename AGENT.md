# Flint — AI Development Guide

Welcome to the Flint codebase.

This repository is developed using an AI-assisted workflow where planning, architecture, and implementation are treated as separate phases.

Before making any code changes, understand the project, review the architecture, and follow the established engineering rules.

# Project

Flint is a proactive, domain-aware Slack AI agent that detects engineering workflow friction and orchestrates actions across multiple productivity tools before users explicitly ask for assistance.

Hackathon

Slack Agent Builder Challenge 2026

Primary Objective

Build a production-quality Slack AI Agent that demonstrates proactive intelligence rather than reactive assistance.

# Development Philosophy

The repository follows four principles.

1. Understand before implementing.
2. Plan before coding.
3. Keep architecture modular.
4. Never sacrifice maintainability for short-term speed.

# Expectations

Before implementing any feature:

- Understand the existing architecture.
- Review related plans.
- Explain your implementation approach.
- Keep modules focused.
- Respect established contracts.
- Update documentation when necessary.

---

# Never

- Refactor unrelated code.
- Introduce unnecessary abstractions.
- Break contracts.
- Ignore existing documentation.
- Add dependencies without justification.
- Make architectural changes without explanation.

---

# Always

- Explain reasoning.
- Keep implementations simple.
- Write maintainable code.
- Prefer readability over cleverness.
- Ask for clarification when requirements are ambiguous.

---

# Success Criteria

Every contribution should improve one or more of the following:

- Maintainability
- Readability
- Reliability
- Extensibility
- Developer Experience

The goal is not only to build Flint but to build it professionally.
