# Flint — Project Glossary

> This document defines the terminology, concepts, and abbreviations used throughout the Flint project.

---

# Purpose

As Flint grows, many project-specific terms will emerge.

This glossary ensures that every contributor shares the same understanding of those terms.

It serves as the common vocabulary for:

- Developers
- AI assistants
- Reviewers
- Future contributors

Whenever a new architectural or product concept is introduced, it should be added to this document.

---

# A

## Action Card

A Slack Block Kit message presenting suggested or completed actions to the user.

---

# B

## Block Kit

Slack's framework for building rich interactive user interfaces inside Slack.

---

# C

## Context

The collection of information gathered from Slack and connected services before Flint performs reasoning or executes actions.

---

## Context Engine

The subsystem responsible for collecting and organizing relevant contextual information.

---

## Confidence Score

A numerical estimate representing Flint's confidence in a recommendation or detected pattern.

Higher confidence should result in more proactive recommendations.

---

# D

## Detector

A specialized module responsible for identifying a specific type of engineering friction.

Examples include:

- Duplicate Bug Detector
- Stale PR Detector
- Release Blocker Detector

---

# E

## Execution Plan

A structured sequence of actions generated by the Planner before interacting with external services.

---

# F

## Fan-Out

The process of executing multiple independent actions in parallel across connected platforms.

Example:

One Slack discussion

↓

Multiple external updates

---

## Flint

The proactive Slack AI agent developed for the Slack Agent Builder Challenge.

---

# I

## Intent

The underlying objective inferred from a user's message or workspace event.

---

## Intent Engine

The reasoning component responsible for understanding user requests and engineering discussions.

---

# M

## MCP

Model Context Protocol.

Provides a standardized interface for interacting with external systems and productivity tools.

---

## Monitoring Loop

A recurring workflow responsible for observing workspace activity and detecting meaningful events.

---

# O

## Orchestration

The coordinated execution of actions across multiple connected systems as part of a single workflow.

---

# P

## Pattern

A recurring engineering event that Flint recognizes as operational friction.

Examples include:

- Duplicate incidents
- Stale discussions
- Blocked releases
- Repeated questions

---

## Pattern Detector

A module specialized in identifying one category of engineering pattern.

---

## Planner

The decision-making component that determines which actions should be executed and in what order.

---

# R

## Reactive Workflow

A workflow initiated directly by a user through mentions, slash commands, or interactive actions.

---

## Redis Memory

Operational storage used for caching, deduplication, and lightweight workspace state.

---

## RTS

Real-Time Search.

Slack capability that allows Flint to continuously retrieve relevant workspace context.

---

# S

## Slack Workspace

The primary operating environment in which Flint observes conversations and interacts with users.

---

# W

## Workspace Memory

Persistent operational knowledge maintained across monitoring cycles.

Examples include:

- Previously detected events
- Notification history
- Cached context
- Configuration

---

# Adding New Terms

Whenever a new concept is introduced into the architecture, product, or engineering workflow, it should be documented here.

This glossary should remain concise, accurate, and alphabetically organized.
